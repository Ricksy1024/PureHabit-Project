## Executive Summary

The implementation establishes a solid modular structure (`core/` vs `handlers/`) and has broad unit coverage with all current tests passing (8/8 suites). However, there are two blocker-level issues: sync conflict resolution is not atomic under concurrent requests (can lose a `completed=true` event), and the sync endpoint does not validate habit ownership at the trust boundary. Together, these violate the feature constitution and prevent approval for shipping.

## Recommended Actions

### Must Fix (Blockers)

1. Make sync conflict resolution atomic with Firestore transactions to preserve Logical OR under concurrency (`functions/src/handlers/api.js`).
2. Enforce habit ownership authorization in `syncHabitLogsHandler` before writing logs (`functions/src/handlers/api.js`).

### Should Fix (Warnings)

1. Standardize timezone source from user profile for sync and streak calculations (`functions/src/handlers/api.js`, `functions/src/handlers/triggers.js`).
2. Reduce scheduler and streak recomputation query load (pagination/batching/incremental updates) (`functions/src/handlers/triggers.js`).
3. Encrypt TOTP secrets at rest and tighten secret-field access (`functions/src/handlers/api.js`, user profile schema).
4. Add missing critical tests for concurrency, authorization boundaries, and trigger/scheduler behavior (`functions/tests/**`).

### Nice to Fix (Suggestions)

1. Merge existing custom claims when setting `totpVerified` to avoid claim clobbering (`functions/src/handlers/api.js`).
